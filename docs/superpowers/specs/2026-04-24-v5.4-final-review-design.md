# v5.4 — Final Review Fixes + Cookbook Alignment

**Datum:** 2026-04-24
**Status:** Spec, nach Brainstorming abgeschlossen
**Zielrelease:** v5.4.0 (Minor, kein Verhaltens-Change für Endnutzer)

## Ziel

Finale Qualitäts-Runde nach v5.3.0. Drei unabhängige Agenten-Reviews (plugin-validator, Cross-Skill-Consistency, Cookbook-Gap-Analyse gegen `anthropics/skills @ 5128e186`) haben 29 Findings ergeben. v5.4 schließt alle — Konventions-Alignment zur offiziellen Anthropic-Skill-Spec, Struktur-Konsistenz über die 13 Skills hinweg, Cleanup unnötiger Artefakte. Danach ist das Plugin auf dem Stand, den ein Release-Audit ohne Abzüge passiert.

## Motivation

v5.3.0 brachte die Memory→Projekt-lokal-Migration, aber beim Review fielen drei Klassen von Altlasten auf:

1. **Konventions-Drift** — alle 13 Skills nutzen `name: Title Case with Spaces`, die offizielle Anthropic-Konvention und alle 17 Cookbook-Skills nutzen kebab-case. Zwei Agents haben keine `tools:`-Allowlist, zwei Commands keinen `disable-model-invocation:`-Guard.
2. **Struktur-Inkonsistenz** — E4 hatte einen Skill-Kanon etabliert (Vorbedingungen, Keine Fabrikation, Abgrenzung, Kontext-Dateien, Core-Workflow, Output-Format, Wichtige Regeln), aber zwei Skills haben keinen `## Abgrenzung`-Block, einer hat hardcoded statt Variant-Selector, der `./`-Prefix für Kontext-Dateien ist 14× gesetzt und 61× nicht.
3. **Repo-Hygiene** — `settings.json` im Root wird von Claude Code nicht geladen, `.mcp.json` ist leer, `templates/` (Root) und `config/scoring.yaml` sind dead code, `docs/superpowers/` enthält persönliche Brainstorming-Artefakte, die nicht ins Release-Repo gehören.

Gleichzeitig empfiehlt das Cookbook Patterns, die wir nur teilweise umgesetzt haben — Few-Shot-Beispiele in allen Skills (nur 4 von 13), `## Übersicht` als erste Section (fehlt durchgängig), `compatibility:`-Frontmatter für API-Dependencies.

## Design-Entscheidungen (aus Brainstorm)

| # | Entscheidung | Begründung |
|---|---|---|
| D1 | **Ein Release v5.4.0** (nicht gesplittet) | Alle Findings aus einer zusammenhängenden Review; ein Changelog; Cleanup hängt von Konventions-Fixes ab. |
| D2 | **Batch-basiert mit 8 thematischen Commits** | Ein Commit pro Batch bündelt verwandte Änderungen, bläht History nicht auf. |
| D3 | **Subagent-driven, multi-Worktree, parallelisiert** | User-Anforderung. 8 Worktrees in 2 Waves + 1 sequenzieller Tree. Konflikt-Vermeidung durch Datei-Domäne, nicht Fix-Typ. |
| D4 | **kebab-case für Skill-Namen, keine Back-Compat-Aliasse** | Plugin ist nicht öffentlich veröffentlicht, keine externen Konsumenten. Saubere Konvention statt Legacy-Last. |
| D5 | **`docs/superpowers/` aus Git-Tracking, aber lokal behalten** | Brainstorming-Artefakte sind persönliche Arbeitsprodukte, nicht Plugin-Release-Inhalt. Wie `.claude/` (bereits gitignored). |
| D6 | **Dead Code konsequent löschen** | `templates/` (Root, 0 Referenzen), `config/scoring.yaml` (obsolet seit E2-Inline-Scoring) raus. |

## H1-Migrations-Strategie

Risiko: Skills referenzieren sich untereinander in **Title Case** in Prosa (z. B. `skills/academic-context/SKILL.md:104` "Citation Extraction, Literature Gap Analysis, Advisor"). Reiner Frontmatter-Wechsel würde Dangling-References erzeugen.

**Strategie:**
1. **Wave 1.B** ändert nur `name:`-Frontmatter-Feld — nichts in Prosa
2. **Wave 2.S\*** macht Cross-Ref-Cleanup zusammen mit Content-Fixes
3. **Baseline-Grep** vor Wave 2: Liste aller Title-Case-Referenzen, Anzahl dokumentiert. Nach Wave 2 muss Grep `0` zurückgeben.
4. **Konvention:** In Prosa wird der Name in Backticks + kebab-case geschrieben (`` `academic-context` ``), nicht in Anführungszeichen.
5. **Ausnahme:** `CHANGELOG.md`-Einträge aus früheren Versionen bleiben unverändert (historische Akkuratheit).
