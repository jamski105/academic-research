## Context
Audit-§7.2 (F4) beschreibt `/humanizer-de` als first-class Workflow in academic-research: `humanizer-de` existiert bereits als globaler Skill unter `~/.codex/skills/humanizer-de/` und muss nicht re-implementiert werden. §18.2 liefert den konkreten Skeleton für `commands/humanize.md`. Ticket #69 implementiert den eigenständigen Slash-Command-Pfad; die Integration in die `chapter-writer`-Pipeline ist Scope von #68. Der Command liest ein Kapitel-File, delegiert an `humanizer-de` (Modus wählbar), und schreibt zwei Output-Files: humanisierte Version + Severity-gegliedertes Diff.

## Acceptance Criteria
- [ ] `commands/humanize.md` existiert mit YAML-Frontmatter (`Skill(humanizer-de)`, `argument-hint: <kapitel-pfad> [--mode normal|deep]`)
- [ ] `/academic-research:humanize kapitel/3.md --mode deep` erzeugt `kapitel/3.humanized.md`
- [ ] `/academic-research:humanize kapitel/3.md --mode deep` erzeugt `kapitel/3.diff.md`
- [ ] `kapitel/3.diff.md` gliedert Änderungen nach Severity-Stufen (entsprechend `humanizer-de`-Skill-Output)
- [ ] Erstaktivierung prüft ob Voice-Samples unter `~/.academic-research/projects/<slug>/voice-samples/` vorliegen; falls nicht, fragt der Command nach Samples
- [ ] Default-Modus ohne `--mode`-Flag ist definiert (entweder `normal` oder `deep`)

## Out of Scope
Integration in `chapter-writer`-Pipeline (draft → humanizer-de → quality-reviewer → final) ist Scope von #68.
